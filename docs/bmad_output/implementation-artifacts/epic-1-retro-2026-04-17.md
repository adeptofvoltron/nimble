## Successes

1. **Architecture document delivered.** Every story had precise "exact required content" sections — pyproject.toml scaffold, HotkeyAdapter ABC, ci.yml. Dev agent had zero ambiguity on toolchain details.

2. **FakeHotkeyAdapter established early.** Placing it in Story 1.2 rather than deferring to when tests first need it means every Epic 2 story has test infrastructure available on day one.

3. **Code review caught real issues.** The `.nimble/manifest.lock` gitignore negation (`.nimble/` → `.nimble/*`) was a functional correctness fix, not cosmetic — it would have silently broken Epic 6 reproducible installs if missed.

4. **Cross-story scope discipline.** "Not created in this story" sections in each dev notes prevented scope creep. Clean handoffs between 1-1 → 1-2 → 1-3.

5. **pytest exit code 5 thread.** The solution emerged in 1-1's review, was documented in 1-2, and became the CI implementation in 1-3. Good example of compounding knowledge across stories.

6. **Smooth delivery.** Simple, well-scoped changes delivered without friction. The simplicity was intentional and correct for a foundation epic.

---

## Challenges

1. **flake8 / pyproject.toml gap.** flake8 does not natively support `pyproject.toml`. We assumed it would pick up config there; it doesn't. A `setup.cfg` patch resolved it, but the toolchain config is now split across two files. Architecture doc did not flag this known flake8 quirk.

2. **pynput headless CI concern deferred three stories running.** Story 1.1 deferred it. Story 1.2 deferred it. Story 1.3 deferred it. This is a forward risk, not a past failure — but Story 2.2 cannot defer it again. `import pynput.keyboard` raises `ImportError` on headless Linux without `$DISPLAY`. CI runs on `ubuntu-latest` with no display server.

3. **`nimble.yaml` vs `config.yaml` inconsistency in architecture doc.** One early diagram uses `nimble.yaml`; everything else uses `config.yaml`. Low risk for now, but a future agent reading the wrong diagram could scaffold the wrong filename.

---

## Key Insights

1. Tight specs with "exact required content" sections eliminate ambiguity without slowing planning. Worth the upfront investment.
2. Deferred items documented in a shared `deferred-work.md` compound — they don't disappear, they accumulate. Each deferred item in Epic 1 is a forward risk for Epic 2.
3. The pynput platform factory pattern (Story 2.1) is the architectural answer to the CI problem, but Story 2.2 must confirm the CI guard implementation before merging.
